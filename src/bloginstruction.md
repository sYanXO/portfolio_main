# blog writing rules

follow these when writing or editing any post on this site.

## voice and tone

- first person, lowercase. casual but not sloppy.
- write like you're explaining something to a friend who's smart but hasn't seen this before.
- be direct. say what you mean, then move on.
- playful is good. sarcasm is fine. but don't try too hard to be funny.
- show your actual thought process. let the reader watch you work through the problem.

## hard rules

- **never use em dashes.** not the unicode character, not double hyphens pretending to be one. if you catch yourself reaching for one, use a period, a comma, a colon, or parentheses instead. restructure the sentence if you have to.
- **never use air quotes in prose.** no wrapping words in quotation marks for emphasis or scare-quoting. if a term needs emphasis, use *italics* or **bold**. quotation marks are only for actual string literals in code context.
- **no exclamation marks** unless something genuinely warrants one (almost nothing does).
- **no filler phrases.** skip things like "it's worth noting that," "interestingly," "as we can see," "in this section we will." just say the thing.

## structure

- start with the problem. no preamble, no throat clearing.
- use short paragraphs. one idea per paragraph.
- headers should be lowercase and describe what happens in that section. headers are not clever.
- code blocks should be minimal. show the idea, not the boilerplate.
- end with either a forward pointer (what's next) or a clean closing thought. no summaries of what the reader just read.

## punctuation and formatting

- periods and commas do most of the work. colons for lists or elaborations. parentheses for asides.
- **bold** for terms being introduced or key phrases.
- *italics* for emphasis or when referencing a concept.
- backticks for code, filenames, commands, and technical identifiers.
- keep sentences short when you can. long sentences are fine if they have a clear rhythm, but don't let them run away from you.

## things to avoid

- explaining what you're about to explain. just explain it.
- hedging language (probably, arguably, it could be said that). commit to what you're saying.
- repeating the same point in different words. trust the reader.
- paragraphs that are just one long sentence. break them up.
- sounding like a textbook. this is a blog, not documentation.
- avoid formulaic contrast lines that sound generated. examples: "this isn't X, it's Y", "the real problem is not X", "there is no X. there is only Y." if the point matters, say it directly in one sentence.
